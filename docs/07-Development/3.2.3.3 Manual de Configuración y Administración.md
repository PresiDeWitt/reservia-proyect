## 1️⃣ Variables de entorno

> [!warning] Archivo `backend/.env` — configuración completa
>
> | Variable | Entorno | Obligatoria | Descripción |
> |---|---|---|---|
> | `SECRET_KEY` | dev / prod | ✅ | Clave criptográfica de Django. Única y aleatoria en producción. |
> | `DEBUG` | dev / prod | ✅ | `True` en desarrollo · **`False` en producción** |
> | `ALLOWED_HOSTS` | dev / prod | ✅ | Hosts permitidos separados por coma |
> | `CORS_ALLOWED_ORIGINS` | dev / prod | ✅ | Orígenes CORS permitidos (URL del frontend) |
> | `OPENROUTER_API_KEY` | dev / prod | ⚠️ | Para el chatbot IA. Sin ella, el chat devuelve 503. |
> | `DATABASE_URL` | prod | ⚠️ | URL PostgreSQL. Si no existe, usa SQLite. |
> | `DB_PATH` | dev/docker | ❌ | Ruta personalizada del archivo SQLite |
> | `SECURE_HSTS_SECONDS` | prod | ❌ | Segundos de HSTS. Default: 31536000 (1 año) en prod. |

> [!tip] Generar una SECRET_KEY segura
> ```bash
> python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
> ```

> [!danger] Reglas de seguridad para producción
>
> | Regla | Estado requerido |
> |---|---|
> | `DEBUG = False` | ✅ Obligatorio |
> | `SECRET_KEY` única y larga | ✅ Obligatorio |
> | `ALLOWED_HOSTS` sin comodín `*` | ✅ Obligatorio |
> | `.env` en `.gitignore` | ✅ Obligatorio |
> | `OPENROUTER_API_KEY` nunca en commits | ✅ Obligatorio |

## 2️⃣ Configuración de base de datos

> [!info] Selección automática de motor
>
> ```
> DATABASE_URL presente en entorno
>         │
>        SÍ ──→ PostgreSQL (dj-database-url)   ← producción Railway
>         │
>         NO ──→ SQLite  (archivo local)        ← desarrollo / Docker
> ```

> [!abstract] Comandos de gestión de BD
>
> ```bash
> # Aplicar migraciones pendientes
> python manage.py migrate
>
> # Ver migraciones y su estado
> python manage.py showmigrations
>
> # Generar nueva migración tras cambios en models.py
> python manage.py makemigrations
>
> # Poblar con datos iniciales
> python manage.py seed
>
> # Limpiar todos los datos (mantiene esquema)
> python manage.py flush --noinput
>
> # Reset completo (solo SQLite)
> rm backend/db.sqlite3 && python manage.py migrate && python manage.py seed
> ```

> [!warning] Migraciones en producción (Railway)
>
> Las migraciones se aplican **automáticamente** en cada deploy.
> El entrypoint del Dockerfile ejecuta:
> ```
> python manage.py migrate → python manage.py seed → gunicorn
> ```
> No es necesario aplicarlas manualmente en Railway.

## 4️⃣ Configuración de rate limiting (throttling)

> [!bug] Archivo `backend/api/throttling.py`
>
> | Clase | Scope | Endpoint protegido |
> |---|---|---|
> | `LoginRateThrottle` | `login` | `POST /api/auth/login/` |
> | `RegisterRateThrottle` | `register` | `POST /api/auth/register/` |
> | `ChatRateThrottle` | `chat` | `POST /api/chat/` |

> [!warning] Cambiar los límites de throttle
>
> En `backend/reservia/settings.py`, bloque `DEFAULT_THROTTLE_RATES`:
>
> ```python
> REST_FRAMEWORK = {
>     'DEFAULT_THROTTLE_RATES': {
>         'login':    '10/minute',   # ← ajusta aquí
>         'register': '5/minute',    # ← ajusta aquí
>         'chat':     '20/hour',     # ← ajusta aquí
>     }
> }
> ```
>
> Formatos válidos: `N/second` · `N/minute` · `N/hour` · `N/day`

## 5️⃣ Configuración de seguridad

> [!danger] Checklist de hardening para producción
>
> | Configuración | Archivo | Valor en prod |
> |---|---|---|
> | `DEBUG` | `settings.py` / `.env` | `False` |
> | `SECURE_SSL_REDIRECT` | `settings.py` | `True` (auto si DEBUG=False) |
> | `SECURE_HSTS_SECONDS` | `settings.py` | `31536000` |
> | `SESSION_COOKIE_SECURE` | `settings.py` | `True` |
> | `CSRF_COOKIE_SECURE` | `settings.py` | `True` |
> | `X_FRAME_OPTIONS` | `settings.py` | `'DENY'` |
> | `SECURE_PROXY_SSL_HEADER` | `settings.py` | `('HTTP_X_FORWARDED_PROTO', 'https')` |
>
> **Verificar hardening:**
> ```bash
> python manage.py check --deploy
> # Esperado: System check identified no issues (0 silenced).
> ```

> [!tip] Auditoría de dependencias
>
> ```bash
> # Backend
> pip install bandit
> bandit -r backend/api backend/reservia -x backend/api/migrations -ll -ii
>
> # Frontend
> cd frontend
> npm audit --omit=dev --audit-level=moderate
> ```

## 6️⃣ CI/CD — GitHub Actions

> [!success] Pipeline automático `.github/workflows/ci-lint-unit-sast.yml`
>
> Se ejecuta en cada **push** o **pull request** a `main`:
>
> | Job | Comando | Timeout |
> |---|---|---|
> | `frontend-lint` | `npm run lint` | 10 min |
> | `frontend-unit-tests` | `npm run test:run` | 15 min |
> | `backend-unit-tests` | `python manage.py test tests` | 15 min |
> | `backend-sast` | `bandit -r backend/api backend/reservia` | 10 min |

> [!abstract] Variables de entorno en CI
>
> El job de backend inyecta automáticamente:
> ```yaml
> env:
>   SECRET_KEY: ci-test-secret-key
>   DEBUG: 'True'
> ```
> No se necesita ninguna configuración adicional en GitHub Secrets para los tests.

## 7️⃣ Gestión del despliegue en Railway

> [!info] Operaciones desde el panel Railway
>
> | Acción | Cómo hacerlo |
> |---|---|
> | Redeploy manual | Railway → Service → **Redeploy** |
> | Ver logs en tiempo real | Railway → Deployments → deployment activo → **Logs** |
> | Editar variables de entorno | Railway → Service → **Variables** |
> | Escalar workers | Modificar `--workers N` en el `Dockerfile` entrypoint |
> | Rollback a deploy anterior | Railway → Deployments → deployment anterior → **Rollback** |
> | Ver métricas | Railway → Service → **Metrics** (CPU, RAM, red) |

> [!tip] CLI de Railway (opcional)
>
> ```bash
> npm install -g @railway/cli
> railway login
> railway link               # conectar al proyecto
> railway variables          # ver variables de entorno
> railway logs               # ver logs en tiempo real
> railway up                 # deploy manual desde CLI
> ```

## 9️⃣ Monitoreo y logs

> [!quote] Qué vigilar en producción
>
> | Indicador | Señal de alerta | Acción |
> |---|---|---|
> | Errores 500 en logs | Fallo en la API | Revisar logs Railway → buscar traceback Python |
> | Muchos 429 en login | Posible ataque de fuerza bruta | Reducir límite en `DEFAULT_THROTTLE_RATES` |
> | `AI service error` en chat | Fallo de OpenRouter | Verificar `OPENROUTER_API_KEY` y estado de OpenRouter |
> | Tiempo de respuesta alto | Carga elevada | Aumentar workers en Gunicorn |
> | Deploy fallido en Railway | Error en el build | Ver logs del build — suele ser dependencia o migración |

> [!success] Logs de arranque esperados (contenedor sano)
>
> ```
> Running migrations...          ✅
> Seeding database...            ✅
> [INFO] Starting gunicorn       ✅
> [INFO] Listening at: http://...✅
> [INFO] Worker booted (pid: X)  ✅  × 3
> ```
