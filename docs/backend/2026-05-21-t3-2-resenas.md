# T3.2 — Sistema de Reseñas

## Problema
El `rating` y `reviews_count` de cada restaurante eran valores fijos del seed. No había forma de que los usuarios dejasen reseñas reales, por lo que el sistema de puntuación carecía de valor real.

## Solución
Modelo `Review` con protección de acceso: sólo pueden opinar usuarios con una reserva confirmada pasada en ese restaurante, y sólo una vez por restaurante. Al crear una reseña se recalcula automáticamente el rating agregado del restaurante.

## Qué se hizo

### Backend
- `api/models.py` — nuevo modelo `Review` (user, restaurant, rating 1-5, comment, created_at; `unique_together` user+restaurant)
- `api/migrations/0004_review.py` — migración generada con `makemigrations`
- `api/serializers.py` — `ReviewSerializer` con campo computed `userName` y validación de rating 1-5
- `api/views.py` — vista `restaurant_reviews` (GET lista + metadatos `can_review`/`has_reviewed`; POST crea con guardia de reserva pasada y recalcula rating)
- `api/urls.py` — `GET|POST /api/restaurants/<pk>/reviews/`
- `management/commands/seed.py` — seed de 5 usuarios demo + 14 reseñas distribuidas; recalcula ratings reales tras insertar

### Frontend
- `api/restaurants.ts` — interfaces `Review`, `ReviewsResponse`; métodos `reviews()` y `createReview()`
- `pages/RestaurantDetails.tsx` — tab "Reseñas" reescrito: puntuación real, lista de reseñas con avatar/nombre/fecha/estrellas, formulario de envío con `StarInput` interactivo (sólo visible si `can_review`), estado `has_reviewed`, shimmer de carga

### Tests
- `tests/test_reviews_api.py` — 10 tests: GET anónimo, `can_review` con/sin reserva, `has_reviewed`, creación, actualización de rating, doble reseña bloqueada, rating inválido

## Cómo
- Vista única `restaurant_reviews` maneja GET y POST con `IsAuthenticatedOrReadOnly`
- La comprobación de elegibilidad es `Reservation.objects.filter(user, restaurant, status='confirmed', date__lt=today)`
- El recálculo de rating usa `Avg` + `Count` de Django ORM sobre los `Review` del restaurante
- El frontend carga las reseñas sólo cuando el usuario abre el tab (lazy), y re-fetch al cambiar de usuario autenticado

## Riesgos
- Los seed reviews usan usuarios demo sin contraseña; no pueden hacer login pero tampoco bloquean nada
- El tab no pagina reseñas — si un restaurante acumula muchas (>50) podría ser lento; aceptable para el TFG
- El rating del restaurante es ahora dinámico; si se elimina una reseña vía admin sin recalcular quedaría desincronizado (no hay endpoint de delete de reseñas)
